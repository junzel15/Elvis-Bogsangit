// 03 workflow

AI-Assisted Development with Claude Code

Pairing with Claude across the full dev loop to ship secure, production-grade mobile and cloud features faster.

Claude Code lives in my terminal next to the repo. I drive it for multi-file refactors across React Native screens and AWS Lambda handlers, generate Jest and integration tests against real fixtures, and debug tricky issues that span the mobile client, API Gateway, and DynamoDB. Project-level CLAUDE.md files encode conventions (folder layout, naming, lint rules, commit style) so suggestions land closer to merge-ready on the first pass. Custom slash commands cover the repeatable work — scaffolding a new screen with navigation and a Redux slice wired up, generating a Lambda with IAM policy and Cognito auth, writing a PR description from the diff. Hooks handle the boring guardrails (format on stop, run tests before commit), and MCP servers connect Claude to project-specific tooling like the local DynamoDB instance or the Figma file driving a screen.

Beyond the editor, I build LLM features directly into product using the Anthropic API — chat surfaces, summarization, structured extraction with tool use, streaming responses to React Native via SSE. Prompt caching keeps long system prompts and shared context cheap when a feature gets real traffic. IDE assistants and AI-driven code review sit in the PR loop to catch obvious issues before a human reviewer spends time on them. The throughput gain is real, but engineering judgment still owns correctness — types, tests, observability, and a human reading the diff before it ships.

In practice

  • Multi-file refactors, test generation, and cross-stack debugging on RN + AWS repos
  • Project-level CLAUDE.md + custom slash commands that encode conventions
  • Hooks & MCP integrations for format-on-save, pre-commit checks, project tooling
  • Anthropic API in production: tool use, structured outputs, streaming, prompt caching
  • AI-assisted PR review loops that catch regressions before human review
  • Deliberate Sonnet vs Opus selection — Sonnet for iteration, Opus for deep work

EF Tech

Next.js 15 commerce site for computers, laptops, printers, and all the parts that keep them running.

A full App Router build with Server Components for every page, Server Actions powering an HTTP-only-cookie cart (no client store needed), and a React Hook Form + Zod checkout flow that clears the cart and writes a status-coded order on submit. Six top-level categories — Computers, Laptops, Printers, plus a deep "Parts" tree for CPUs, GPUs, batteries, toners, drums, and more — with a 32-product mock catalog covering Dell, HP, Lenovo, Apple, ASUS, NVIDIA, Brother, Epson, and Canon.

Catalog filters drive the URL: category, brand, price tier, and in-stock state all read from searchParams and filter server-side, so deep links and back-button history behave correctly. Product detail pages stream gallery, spec table, and stock badge from typed mock data, with a sticky "customers also bought" rail. Tailwind design system keeps cards, buttons, and chips consistent across the whole funnel.
